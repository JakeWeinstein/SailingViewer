<task type="auto">
  <name>Task 1: Rewrite VideoWatchView with YT.Player + chapter list + auto-advance + mobile</name>
  <files>components/VideoWatchView.tsx, app/layout.tsx</files>
  <action>
1. In `app/layout.tsx`, add a client-side call to `loadYouTubeAPI()` so the YouTube IFrame API script is loaded once globally. This can be done with a small client component wrapper or a `<Script>` tag approach — but the `onYouTubeIframeAPIReady` callback queue in `lib/youtube-api.ts` must be the one that fires. The simplest approach is to import and call `loadYouTubeAPI()` from a `'use client'` component rendered in the layout (e.g., `<YouTubeLoader />`).

2. Rewrite `components/VideoWatchView.tsx` to use `new window.YT.Player()` for ALL video rendering (not raw `<iframe src="...">` tags). Key changes:

   **Player initialization:**
   - Use `onYouTubeReady()` from `lib/youtube-api.ts` to wait for API
   - Create `ytPlayerRef = useRef<YT.Player | null>(null)`
   - In a useEffect keyed on `[videoId]`, call `onYouTubeReady(() => { ... new window.YT.Player(containerId, { videoId, playerVars, events }) })`
   - playerVars: `{ playsinline: isMobile ? 0 : 1, rel: 0, fs: 1, start: startSeconds ?? 0 }`
   - Detect mobile: `window.innerWidth <= 640`
   - On cleanup: `ytPlayerRef.current?.destroy()`

   **onStateChange for auto-advance (VID-04):**
   - When `e.data === window.YT.PlayerState.ENDED` and chapters exist and current chapter is not the last:
     - Advance to next chapter
     - If next chapter has a different video ID (multi-part): load new video via `ytPlayerRef.current.loadVideoById(nextVideoId, nextStartSeconds)`
     - If same video ID: just `seekTo(nextStartSeconds)`
   - Track `activeChapterIndex` in state

   **Chapter navigation as vertical list (VID-03):**
   - Replace any pill-based chapter layout with a vertical scrollable list
   - Each chapter entry: timestamp (mono font), title, optional description
   - Active chapter highlighted with `bg-purple-600 text-white` (per Research code example)
   - Clicking a chapter:
     - If same video: `ytPlayerRef.current.seekTo(ch.start_seconds, true)`
     - If different video (multi-part): `ytPlayerRef.current.loadVideoById(ch.video_ref, ch.start_seconds)`
   - Use `clsx` for conditional active styling
   - `max-h-48 overflow-y-auto` for scrollable list with many chapters

   **Mobile layout (VID-05):**
   - Desktop: video left (65%) + sidebar right (35%) in a flex row — keep existing modal pattern
   - Mobile (< 640px): stack vertically — video on top, sidebar (chapters + comments) below
   - Use Tailwind responsive: `flex flex-col sm:flex-row`
   - On mobile after player onReady: attempt `document.getElementById(containerId)?.requestFullscreen()` in a try-catch (works on Android, silently fails on iOS per Research Pitfall 3)

   **Timestamp auto-capture for comments:**
   - When user focuses the comment textarea, if no timestamp is already set:
     - Check `ytPlayerRef.current.getPlayerState()` — only capture if PLAYING (1) or PAUSED (2)
     - Call `Math.floor(ytPlayerRef.current.getCurrentTime())`
     - Pre-fill timestamp with `formatTime(seconds)`
   - User can clear the timestamp field (the "x" button already exists or add one)
   - User can manually edit the timestamp

   **Remove all Drive-specific code paths:**
   - Remove `embedUrl()` usage — replace with YouTube player
   - Remove `videoType === 'drive'` conditionals
   - Remove any `<iframe src="...drive.google.com...">` rendering
   - Remove postMessage-based state detection (replaced by onStateChange events)

3. Keep existing functionality that still applies:
   - Modal overlay with backdrop click + Escape to close
   - Comment list rendering and posting
   - Captain note display and inline editing
   - Favorite toggle button
